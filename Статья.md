# Как мы скрыли граф социальных связей в мессенджере: токены, cover traffic и Tor

Сквозное шифрование — действительно отличная технология, позволяющая обмениваться информацией без страха за конфиденциальность содержимого. Однако оно не защищает от анализа метаданных. Это понятно: E2EE — способ шифрования информации, но не скрытие трафика, времени отправки, адресата и адресанта.

## Проблема метаданных

Представьте: вы используете мессенджер с E2EE. Содержимое сообщений зашифровано и сервер его не видит. Но сервер всё равно знает:

- Кто с кем общается (fingerprint отправителя → fingerprint получателя)
- Когда именно происходит общение
- Как часто
- Примерный объём сообщений

Этого достаточно чтобы построить **граф социальных связей** — карту того, кто с кем контактирует. Для журналиста, связанного с источником, или активиста, координирующего акцию — это критическая уязвимость даже при идеальном шифровании содержимого.

Стандартный пакет в большинстве мессенджеров выглядит так:

```json
{
  "type": "message",
  "from": "A1B2C3D4E5F6A7B8",
  "to":   "9F8E7D6C5B4A3210",
  "text": "<зашифровано>"
}
```

Сервер видит пару `from → to` при каждой отправке. Накопив логи, можно восстановить полный граф общения.

## Анонимная маршрутизация по токенам

В Beacon мы реализовали другой подход: сообщения адресуются не по fingerprint пользователя, а по **одноразовым случайным токенам**.

### Как это работает

При подключении клиент генерирует пул из 50 случайных 128-битных токенов и регистрирует их на сервере:

```kotlin
fun ensureMyTokenPool(ctx: Context): List<String> {
    val existing = getMyTokens(ctx)
    if (existing.size >= POOL_SIZE / 2) return existing
    val needed = POOL_SIZE - existing.size
    val newTokens = (1..needed).map { generateToken() }
    // ...сохраняем и возвращаем
}

private fun generateToken(): String {
    val bytes = ByteArray(16)
    rng.nextBytes(bytes)
    val sb = StringBuilder(32)
    for (b in bytes) {
        val v = b.toInt() and 0xff
        if (v < 16) sb.append('0')
        sb.append(v.toString(16))
    }
    return sb.toString()
}
```

Сервер хранит только маппинг `токен → WebSocket`, без привязки к identity пользователя:

```python
token_to_ws:  dict = {}   # token → websocket
ws_to_tokens: dict = {}   # websocket → set[str]
known_tokens: set  = set() # токены которые хоть раз регистрировались
```

Клиенты обмениваются токенами через E2EE — отправляют друг другу пул токенов зашифрованным системным сообщением. После обмена пакет выглядит так:

```json
{
  "type": "anon_message",
  "token": "e39f013483715c300e9a7db2b601709c",
  "payload": "<зашифровано>"
}
```

Сервер не знает кто отправитель и кто получатель. Он просто находит WebSocket по токену и доставляет. Токен одноразовый — после доставки удаляется.

### Первый контакт: анонимный mailbox

Классическая проблема токенной маршрутизации — первое сообщение. Чтобы обменяться токенами, нужно сначала как-то связаться. В большинстве мессенджеров это означает: сервер видит `A → B` хотя бы один раз.

Мы решили это через **анонимный mailbox**, встроенный прямо в инвайт-код.

Инвайт-код в Beacon (формат v3) содержит не только публичный ключ и fingerprint, но и случайный 16-байтный `mailbox_tag`:

```
[version=0x03][timestamp][nonce][mailbox_tag(16б)][fingerprint][ec_point][name][sig]
```

Схема работы:

```
Alice создаёт инвайт → внутри mailbox_tag T
Bob принимает инвайт → шифрует первое сообщение ключом Alice → кладёт в mailbox[T]
Alice при подключении → спрашивает сервер: "есть ли что-то для тегов [T, X, Y, Z, ...]?"
Сервер → возвращает блобы по совпавшим тегам
Alice → расшифровывает, получает сообщение + токены Bob'а
```

Ключевой момент: Alice запрашивает **всегда ровно 20 тегов** — свой настоящий плюс случайные фейковые. Сервер не знает какой из них реальный, и не может по размеру запроса определить сколько у Alice ожидаемых контактов:

```kotlin
fun buildFetchTagList(ctx: Context): List<String> {
    val real = getMyMailboxTags(ctx)
    val fakeCount = maxOf(MBOX_TOTAL - real.size, 0)  // всегда итого 20
    val fakes = (1..fakeCount).map { generateToken() }
    return (real + fakes).shuffled()
}
```

Сервер при получении `mailbox_fetch` возвращает только те теги у которых есть блобы, и сразу их удаляет. Блобы без запроса автоматически удаляются через 48 часов.

Таким образом **первое сообщение больше не раскрывает пару** `A → B`. Сервер видит анонимный блоб и запрос из 20 тегов — и не знает кто кому пишет.

### Доверие к ключу

Есть и более фундаментальный вопрос: при первом контакте клиенты обмениваются публичными ключами. Сам по себе публичный ключ не секрет — но откуда вы знаете что получили именно ключ собеседника, а не ключ сервера-посредника? Теоретически враждебный сервер мог подменить ключ и читать всё шифрование.

В Beacon это решается через **fingerprint** — SHA-256 хэш публичного ключа, который пользователи сверяют **вне системы**: показывают экраны друг другу, зачитывают по телефону, сравнивают в другом мессенджере. Если fingerprint совпал — ключ подлинный и MITM исключён. Это та же концепция что "Safety numbers" в Signal.

```
Сверили fingerprint вне системы
    → ключ подлинный
    → шифрование надёжно
    → токены и mailbox-сообщения защищены
    → всё дальнейшее общение анонимно
```

Если fingerprint не сверяли — вся последующая защита теряет смысл. Это фундаментальное требование любого E2EE мессенджера, и Beacon не исключение.

### Что видит сервер теперь

До: `A1B2C3D4 → 9F8E7D6C (сообщение)`  
После: `anon_message → токен e39f0134... (доставлено)`

Граф социальных связей недоступен для **утёкших логов сообщений**. Однако честно признаем: сервер хранит маппинг `токен → WebSocket`, поэтому активно враждебный оператор сервера всё равно может восстановить кто с кем общается — просто по внутренней таблице маршрутизации. Токены защищают от пассивной утечки логов, но не от намеренно недобросовестного сервера.

Именно поэтому в Beacon поддерживается самохостинг — пользователь может поднять собственный сервер и не зависеть от доверия к оператору.

## Cover Traffic: скрытие по времени

Даже без fingerprint остаётся timing-атака. Если наблюдатель контролирует трафик обоих абонентов, он может статистически сопоставить: "Алиса отправила пакет в 14:23:01.337, Боб получил пакет в 14:23:01.891 — разница 554 мс, совпадает 1000 раз из 1000 → они общаются".

### Случайный шум — слабая защита

Первое, что приходит в голову — отправлять фейковые сообщения случайно, раз в 30–120 секунд. Это создаёт шум, но не решает проблему. На графе трафика такой подход выглядит как редкие пакеты, уходящие на сервер без ответа — на фоне стабильного обмена сообщениями между двумя пользователями они легко фильтруются статистически.

### Постоянный поток — настоящая защита

Правильное решение — **constant-rate cover traffic**, именно так работает академический протокол Loopix. Идея: клиент всегда шлёт поток с фиксированным интервалом. Когда есть реальное сообщение — оно просто занимает место очередного шумового пакета в потоке.

```
Без фичи:  ────M──────────M──M──────────  (M = сообщение, видно паттерн)
С фичей:   N─N─M─N─N─N─N─M─NM─N─N─N─N─  (N = шум, всё выглядит одинаково)
```

Реализация: реальные сообщения ставятся в очередь, корутина с фиксированным интервалом берёт из неё следующий пакет — или отправляет шум если очередь пуста:

```kotlin
private val outboundQueue = LinkedBlockingQueue<String>()

private fun startCoverTraffic() {
    val intervalMs = if (mode == AGGRESSIVE) 1_000L else 5_000L
    coverTrafficJob = scope.launch(Dispatchers.IO) {
        val rng = SecureRandom()
        while (isActive) {
            delay(intervalMs)
            if (!isConnected) continue
            val packet = outboundQueue.poll()
            if (packet != null) {
                webSocket?.send(packet)  // реальное сообщение
            } else {
                webSocket?.send(buildNoisePacket(rng))  // шум
            }
        }
    }
}

private fun sendWs(json: String) {
    if (mode != OFF && isConnected) {
        outboundQueue.offer(json)  // в очередь, не сразу
    } else {
        webSocket?.send(json)
    }
}
```

Сервер тихо дропает шумовые пакеты — токен не в `known_tokens`:

```python
async with lock:
    is_known = token in known_tokens
if not is_known:
    # cover traffic — дропаем без очереди и без лога
    if msg_id:
        await send_safe(websocket, json.dumps({"type": "ack", "id": msg_id}))
    continue
```

В приложении это опциональный режим — пользователь выбирает уровень:

| Режим | Интервал | Трафик |
|---|---|---|
| Выключен | — | только реальные сообщения |
| Умеренный | 1 пакет / 5 сек | ~2–3 МБ/час |
| Агрессивный | 1 пакет / сек | ~20–30 МБ/час |

При включении показывается предупреждение с расходом трафика и батареи — фича явно не для обычных пользователей.

### Честно об ограничениях cover traffic

Постоянный поток скрывает timing при наблюдении **с одной стороны** (например, ISP только отправителя). Если наблюдатель контролирует трафик у обоих абонентов одновременно — статистическая корреляция всё ещё возможна, просто значительно сложнее. Полностью решить эту проблему без mix network (Nym, Loopix) с намеренными задержками и перемешиванием пакетов невозможно. Но это уровень целенаправленной операции спецслужб против конкретного человека — не массового пассивного прослушивания.

## Tor Hidden Service: скрытие IP

Анонимные токены скрывают граф связей от сервера. Но сервер всё ещё видит IP адрес клиента при подключении. Для аудитории которой важна анонимность — это недопустимо.

Решение — поддержка Tor. Но обычный Tor через exit-ноды имеет проблему: Cloudflare и многие хостинги блокируют exit-ноды. Мы настроили **Tor Hidden Service** — `.onion` адрес:

```
amqvpheooju3fg7tafxkmf73c3vg4xg7nycelepiie6jdjzbsqrvrcqd.onion
```

Трафик идёт полностью внутри сети Tor — никаких exit-нод, никакого Cloudflare в цепочке. Конфигурация на сервере минимальная:

```
HiddenServiceDir /var/lib/tor/beacon/
HiddenServicePort 80 127.0.0.1:9000
```

Важный нюанс при реализации на Android: Orbot работает в двух режимах — SOCKS прокси (порт 9050) и системный VPN. В VPN режиме явный SOCKS прокси создаёт конфликт двойного туннелирования. Решение — для `.onion` адресов использовать обычный HTTP клиент, Orbot перехватит трафик на уровне системы сам:

```kotlin
// Onion через Orbot VPN — не используем явный SOCKS прокси
val client = if (wsUrl.contains(".onion")) wsClient else activeWsClient()
webSocket = client.newWebSocket(request, listener)
```

## Итоговая таблица угроз

| Наблюдатель | Видит без защиты | Видит с Beacon |
|---|---|---|
| Честный сервер / утёкшие логи | Граф `from → to`, содержимое | Только анонимные блобы и токены без привязки к identity |
| Враждебный сервер | Граф `from → to`, содержимое | Таблица токенов (решение — самохостинг) |
| ISP отправителя | IP сервера, время и объём | Зашифрованный Tor трафик |
| ISP получателя | IP сервера, время и объём | Зашифрованный Tor трафик |
| Оба ISP одновременно | Timing корреляция | Статистически затруднена (constant-rate) |
| Целевая операция (MITM, устройство) | Всё | Содержимое защищено E2EE |

## Итог

Четыре слоя, каждый закрывает свой вектор:

1. **Анонимный mailbox** — первый контакт не раскрывает пару `A → B`, сервер видит только блобы по случайным тегам
2. **Токены** — все последующие сообщения маршрутизируются без fingerprint
3. **Constant-rate cover traffic** — timing-корреляция затруднена, наблюдатель видит равномерный поток
4. **Tor hidden service** — IP адрес скрыт, нет exit-нод и блокировок Cloudflare

Это не mix network и не абсолютная анонимность. Но для защиты от пассивного массового наблюдения и логов оператора — практически достаточно.

Beacon — открытый проект с возможностью самохостинга. Исходный код: [github.com/MoRoKonst/beacon-messenger](https://github.com/MoRoKonst/beacon-messenger)
