## Модель угроз

Beacon разработан для противодействия следующим типам нарушителей:

### В области действия защиты

| Нарушитель | Возможности | Защита |
|---|---|---|
| Сетевой атакующий | Перехват и модификация трафика | TLS + привязка сертификата + E2EE |
| MITM с поддельным CA | Установить пользовательский CA, проксировать TLS | Обнаружение вторжения (USER_CA + PROXY) |
| Компрометация сервера | Оператор сервера читает БД | E2EE; сервер не хранит открытый текст |
| Кража физического устройства | Офлайн-извлечение файловой системы | EncryptedSharedPreferences + слой SMK |
| Форензический анализ | Cold-boot, JTAG, программное извлечение | SMK требует пароль; режим-приманка |
| Принуждение (юридическое требование) | Принудительная расшифровка | Пароль-паника, уничтожение данных, режим-приманка |
| Отладка / динамический анализ | Подключение JDWP, logcat | Антиотладка, подавление логов в Параноидном режиме |
| Атакующий на рутованном устройстве | Читать преференции от root, обходить keystore | Слой PBKDF2 SMK; предупреждение об обнаружении root |

### Вне области действия защиты

| Сценарий | Причина |
|---|---|
| Полная компрометация конечной точки (root + дамп RAM в реальном времени) | SMK находится в памяти пока приложение разблокировано |
| Корреляция трафика на стороне сервера | Временны́е метки и метаданные в сети |
| Принуждение получателя | Устройство получателя вне контроля |
| Атака на цепочку поставок (вредоносная сборка) | Требует независимой верификации APK |

## Криптографическая схема

### Иерархия ключей

```
Пароль пользователя
    │
    ▼ PBKDF2-SHA256 (300 000 итераций, соль 32 байта)
Ключ, выведенный из пароля (PDK)
    │
    ▼ AES-256-GCM
enc_smk_pwd  ──────────────────────────────────┐
                                               │ расшифровка
AndroidKeyStore AES-256 ("beacon_smk_wrap")    │
    │                                          │
    ▼ AES-256-GCM                              ▼
enc_smk_ks  ──────────────────► Storage Master Key (SMK)  [в памяти]
                                               │
                    ┌──────────────────────────┼──────────────────────┐
                    ▼                          ▼                      ▼
           EC identity privkey          Групповые AES-ключи    Блобы сообщений
         (beacon_ec_keys_enc)               (groups)      (chat_storage_encrypted)
```

### Identity-ключи

Каждый пользователь имеет одну ключевую пару EC на кривой P-256 (secp256r1). Эта пара выступает:
- Долгосрочным identity-ключом в X3DH (отправитель и получатель)
- Ключом подписи для инвайт-кодов и challenge аутентификации

Приватный ключ хранится в `EncryptedSharedPreferences("beacon_ec_keys_enc")` под ключом `"ec_priv"`. Формат хранения:
- `"smk1:" + Base64(iv[12] + AES-256-GCM(privKey.encoded, smk))` — когда SMK настроен
- `Base64(privKey.encoded)` — для устаревших записей (принудительно переоборачивается при первом чтении, если SMK разблокирован)

Публичный ключ хранится как обычный Base64 и передаётся на сервер при регистрации.

### Шифрование сообщений

Сообщения один-на-один шифруются с помощью AES-256-GCM:

```
открытый текст
    │
    ▼ AES/GCM/NoPadding
    │  ключ:  сессионный ключ (из Double Ratchet)
    │  iv:    12 случайных байт (из Cipher.init)
    │  тег:   128-битный тег аутентификации (добавляется в конец)
    ▼
iv[12] || шифртекст || тег[16]
```

Блоб шифртекста передаётся в поле `"data"` JSON-сообщения, закодированный в Base64.

### Прямая секретность

Начальное согласование ключей по схеме X3DH:

1. Получатель публикует на сервере: `IK_B` (identity key), `SPK_B` (signed prekey), `OPK_B` (one-time prekey).
2. Отправитель получает пакет и вычисляет:
   ```
   DH1 = ECDH(IK_A, SPK_B)
   DH2 = ECDH(EK_A, IK_B)       ← EK_A: эфемерный ключ, удаляется после отправки
   DH3 = ECDH(EK_A, SPK_B)
   DH4 = ECDH(EK_A, OPK_B)      ← OPK_B: одноразовый, удаляется сервером после выдачи
   master_secret = KDF(DH1 || DH2 || DH3 || DH4)
   ```
3. `master_secret` инициализирует Double Ratchet.

Каждое последующее сообщение продвигает ratchet. Компрометация одного сессионного ключа не компрометирует прошлые или будущие ключи.

Сервер уведомляет клиента, когда запас OPK опускается ниже порога (`OPK_LOW_WATERMARK = 5`) и предлагает загрузить новые пакеты.

### Обмен ключами группы

Каждая группа имеет 256-битный случайный ключ AES (`groupKey`). Он распределяется каждому участнику по отдельности:

1. Администратор вызывает `generateGroupKey()` → `SecureRandom().nextBytes(32)`.
2. Для каждого участника M:
   ```
   shared = ECDH(adminPriv, memberPub)
   encKey = AES-256-GCM(groupKey, KDF(shared))
   ```
3. Пары `{memberId, encKey}` отправляются на сервер в сообщении `create_group`; сервер доставляет каждый `encKey` только соответствующему участнику.
4. Участник вызывает `decryptGroupKey(encKey, myPriv)` → `groupKey`.

Сообщения группы:
```
открытый текст
    │
    ▼ AES/GCM/NoPadding (ключ = groupKey, iv = случайные 12 байт)
iv[12] || шифртекст || тег[16]
```

При исключении участника администратор генерирует новый `groupKey` и распределяет его оставшимся участникам.

При локальном хранении `groupKey` оборачивается SMK (префикс `"smk1:"`) если SMK разблокирован, иначе хранится как обычный Base64.

### Двойное шифрование на устройстве (SMK)

Все чувствительные данные на устройстве получают второй слой шифрования поверх `EncryptedSharedPreferences`.

#### Генерация SMK

При первой регистрации:
```kotlin
val smk  = SecureRandom().nextBytes(32)   // 256-битный случайный ключ
val salt = SecureRandom().nextBytes(32)   // 256-битная случайная соль

enc_smk_pwd = AES-256-GCM(smk, PBKDF2(password, salt, 300_000, 256))
enc_smk_ks  = AES-256-GCM(smk, keystoreKey)   // AndroidKeyStore AES-256
```

Оба шифртекста хранятся в `EncryptedSharedPreferences("smk_config")`.

#### Пути разблокировки SMK

| Путь | Как | Когда |
|---|---|---|
| Вход по паролю | PBKDF2(пароль, соль, 300 000) → расшифровать `enc_smk_pwd` | Каждый вход |
| Биометрический вход | Ключ AndroidKeyStore → расшифровать `enc_smk_ks` | Успех биометрии |
| Разблокировка после блокировки | Те же два пути | После таймаута / выключения экрана |

Операции PBKDF2 выполняются на `Dispatchers.IO` — никогда на Main-потоке.

#### Блокировка SMK

`StorageKeyManager.lock()` заполняет нулями байтовый массив `smk` в памяти и устанавливает его в `null`. Вызывается:
- Перед `isAppLocked = true` (выключение экрана, таймаут автоблокировки)
- Как первая операция в `WipeManager.wipe()`

#### Оборачивание значений

```
wrapBytes(bytes):
    iv = случайные 12 байт
    ct = AES-256-GCM(bytes, smk, iv)
    вернуть "smk1:" + Base64(iv || ct || тег)

unwrapBytes(stored):
    если stored не начинается с "smk1:":
        вернуть Base64.decode(stored)      # путь совместимости с устаревшими данными
    blob = Base64.decode(stored[5:])
    iv = blob[0:12]
    ct = blob[12:]
    вернуть AES-256-GCM-decrypt(ct, smk, iv)
```

Прозрачная миграция: существующие незащищённые значения читаются как обычный Base64 без ошибок. При следующей записи значение оборачивается. EC приватный ключ использует принудительное переоборачивание: если `StorageKeyManager.isUnlocked`, но сохранённое значение не имеет префикса `"smk1:"`, оно немедленно переоборачивается при первом чтении.

#### Что НЕ оборачивается SMK

| Данные | Причина |
|---|---|
| SPK / OPK приватные ключи | Необходимы при холодном старте до входа |
| Состояния сессий Double Ratchet | Необходимы при холодном старте до входа |

### Хеширование паролей

Пароли приложения хешируются перед хранением:

```
v2:<saltB64>:<hashB64>
  соль = SecureRandom().nextBytes(16)
  хеш  = PBKDF2WithHmacSHA256(пароль, соль, итерации=100_000, длина=256)
```

При входе хранимый хеш сравнивается со свежевыведенным. Автоматическая миграция с устаревшего `SHA-256(password)` (формат без префикса `"v2:"`) выполняется при первом успешном входе.

### Шифрование бэкапа

Файлы бэкапа используют ту же схему с пользовательским паролем бэкапа:

```
соль    = SecureRandom().nextBytes(16)
ключ    = PBKDF2-SHA256(паролькопии, соль, 100_000, 256)
iv      = SecureRandom().nextBytes(12)
payload = AES-256-GCM(сериализованные_данные, ключ, iv)
файл    = соль[16] || iv[12] || payload
```

## Аутентификация

### Server Handshake

Аутентификация — challenge-response через ECDSA. Сервер никогда не хранит и не передаёт пароль:

1. Клиент подключается; сервер отправляет `{type: "challenge", data: "<32 случайных байта в base64>"}`.
2. Клиент подписывает байты challenge своим EC приватным ключом (SHA256withECDSA).
3. Клиент отправляет `{type: "register", username, public_key, signature, ...}`.
4. Сервер верифицирует подпись по публичному ключу.
5. При успехе: `{type: "auth_ok"}` + `{type: "turn_config", ...}`.

Таймаут handshake: 15 секунд.

Конфликт сессий: когда второе устройство аутентифицируется под тем же именем пользователя, сервер отправляет `{type: "session_conflict"}` существующей сессии. Существующий клиент отключается и уведомляет пользователя.

### Инвайт-коды

Инвайт-коды — ECDSA-подписанные deep links:

```
beacon://invite?key=<pubKeyB64>&fp=<fp8B>&nonce=<16символов>&name=<nameB64>&ts=<unix>&sig=<sigB64>
```

Подписываемая строка: `key|fp|nonce|name|ts` (разделитель — символ `|`).
Алгоритм подписи: SHA256withECDSA с identity-ключом приглашающего.
TTL: 7 дней от `ts`.

Верификация:
1. Декодировать `key` → EC публичный ключ.
2. Пересчитать `fp` = первые 8 байт SHA-256(encoded публичный ключ), сравнить.
3. Проверить `ts` + 7 дней > сейчас.
4. Верифицировать ECDSA подпись.

## Антифорензика

### Параноидный режим

Когда включён:
- Все вызовы `Log.d`, `Log.i`, `Log.w` подавляются через обёртку `BLog` (проверяет `ParanoidMode.isEnabled`).
- Выполняется `logcat -c` для очистки буфера logcat текущего процесса.
- `IntrusionDetector.scan()` запускается при каждом `onResume()` (на `Dispatchers.IO`).
- При обнаружении угрозы (`handleThreat()`):
  1. Logcat очищается повторно.
  2. HTTP POST на настроенный пользователем URL-оповещения (fire-and-forget coroutine).
  3. Если включён "wipe при взломе": `WipeManager.wipe(level)`.
  4. Иначе: режим стелс (показать `DecoyScreen`).

### Защита экрана

`WindowManager.LayoutParams.FLAG_SECURE` применяется к главному окну активности. Это предотвращает:
- Скриншоты через системное сочетание клавиш.
- Захват содержимого экрана приложениями записи.
- Отображение миниатюры приложения в экране "Последние приложения".

### Антиотладка

Проверяется при запуске:
- `android.os.Debug.isDebuggerConnected()` — обнаруживает подключённый JDWP-отладчик.
- Проверка порта JDWP через сокет.

Эти проверки выполняются до показа UI. При обнаружении отладчика приложение может завершиться или подавить чувствительные операции.

### Обнаружение root

Сканирует признаки компрометации устройства:
- Распространённые бинарники в PATH: `su`, `busybox`, `magisk`, `daemonsu`.
- Известные имена пакетов Superuser APK.
- Пути записи, которые должны быть доступны только для чтения на стоковой прошивке.
- Эвристики в стиле RootBeer.

Обнаруженный root считается предупреждением; пользователь уведомляется, но принудительного выхода нет. В Параноидном режиме обнаружение root запускает `handleThreat()`.

### Обнаружение вторжения

`IntrusionDetector.scan()` возвращает `ScanResult` со списком активных угроз:

| Угроза | Серьёзность | Описание |
|---|---|---|
| `PROXY` | Средняя | Настроен системный HTTP-прокси |
| `USER_CA` | Средняя | Установлен пользовательский корневой CA |
| `PROXY + USER_CA` | Критическая | Почти достоверный MITM-прокси |
| `VPN` | Низкая | Активно VPN-соединение |
| `ADB` | Средняя | Android Debug Bridge включён |
| `DEV_OPTIONS` | Низкая | Режим разработчика включён |

В Параноидном режиме результат сканирования с критической серьёзностью запускает `handleThreat()`.

### Привязка сертификата

`CertificatePinner` (OkHttp3) привязывает TLS-сертификат сервера или публичный ключ. Соединение, предъявляющее неожиданный сертификат, отклоняется даже если он подписан доверенным CA. Это нейтрализует атаки с установкой пользовательского корневого CA.

## Механизмы экстренного реагирования

### Уровни уничтожения данных

| Уровень | Операции | Применение |
|---|---|---|
| `SOFT` | Обнулить ключи в памяти и состояния сессий | Временная блокировка экрана |
| `HARD` | Удалить все преференции, файлы, базы данных, данные WebView, ключи AndroidKeyStore | Обнаруженная угроза, пересечение границы |
| `NUCLEAR` | `HARD` + `ActivityManager.clearApplicationUserData()` | Максимальная срочность; процесс завершается системой |

`StorageKeyManager.lock()` вызывается как самая первая операция любого уничтожения данных — SMK обнуляется из памяти до любого другого деструктивного шага.

### Пароль-паника

Пользователь может настроить дополнительный "пароль-паника" в настройках. Если при входе введён этот пароль вместо реального:
1. В фоне запускается `WipeManager.wipe(HARD)`.
2. Может отображаться фиктивный экран загрузки для выигрыша времени.

Это позволяет незаметно запустить уничтожение данных, выглядя как обычный вход.

### Мёртвая рука

Запланированный будильник (`AlarmManager.setExactAndAllowWhileIdle`) срабатывает, если пользователь не подал признак жизни в течение настроенного интервала (часы или дни). По будильнику:
1. `WipeReceiver` получает `DMS_FIRE`.
2. Вызывается `WipeManager.wipe()`.

Нормальная отметка о присутствии: пользователь открывает приложение (автоматически) или нажимает кнопку ручной отметки. Временна́я метка последней отметки хранится в `EncryptedSharedPreferences`.

`BootReceiver` восстанавливает будильник DMS после перезагрузки устройства.

### Режим-приманка

Когда уничтожение `HARD` запускается при включённом "режиме-приманке":

1. Перед удалением сохраняется в `beacon_recovery` (открытый файл):
   - `username`
   - `password_hash` (хеш, не сам пароль)
   - `user_id`
   - Случайная выборка имён контактов

2. Выполняется полное уничтожение `HARD`.

3. При следующем запуске приложения обнаруживается `beacon_recovery`:
   - Создаётся фиктивный аккаунт с сохранёнными учётными данными.
   - Заполняются фиктивные записи чатов.
   - Приложение выглядит нормально при поверхностном осмотре.

Фиктивный аккаунт не может расшифровать никакие реальные сообщения (все ключи уничтожены). Обеспечивает правдоподобное отрицание при принуждении.

## Безопасность сессий

- Одна активная сессия на устройство на имя пользователя.
- `device_id` — UUID, генерируется один раз при установке и передаётся при каждой аутентификации.
- Если два устройства аутентифицируются под одним именем, сервер отправляет `session_conflict` старой сессии, которая затем отключается и уведомляет пользователя.
- Токены сессий не хранятся; полная ECDSA-реаутентификация происходит при каждом WebSocket-подключении.

## Известные ограничения

1. **Метаданные на сервере.** Сервер знает, кто с кем общается (социальный граф) и временны́е метки сообщений. Содержимое сообщений ему неизвестно. Защита от анализа трафика не предусмотрена.

2. **SMK в памяти.** Пока приложение разблокировано и находится на переднем плане, SMK живёт в памяти. Полный дамп RAM с рутованного или взломанного устройства может его извлечь.

3. **Программные ключи AndroidKeyStore.** На устройствах без аппаратного Secure Element ключи AndroidKeyStore хранятся как программные ключи в TEE или в `/data`. Атакующий с глубоким доступом к ОС может их извлечь, обходя путь `enc_smk_ks`. Путь `enc_smk_pwd` (PBKDF2 300K) обеспечивает защиту независимо от Keystore.

4. **Истощение OPK.** Если запас OPK пользователя исчерпан и клиент офлайн, сервер может выдать один и тот же OPK нескольким отправителям. Клиенты должны проактивно загружать новые пакеты OPK.

5. **Распределение ключей группы через сервер.** Скомпрометированный сервер может отказаться доставлять новый групповой ключ участнику после ротации, фактически молча исключая его. Для критически важных сценариев рекомендуется внеполосная верификация состава группы.

6. **Пароль бэкапа.** Пароль бэкапа нигде не хранится. Если он утерян, бэкап не может быть расшифрован. Механизма восстановления нет.

7. **Форензика режима-приманки.** Опытный форензический эксперт может обнаружить следы предшествующего уничтожения данных (временны́е метки файловой системы, записи журнала, паттерны wear-leveling на флеш-памяти). Режим-приманка обеспечивает социальное/юридическое прикрытие, а не техническую необнаруживаемость.
