# 🔐 Безопасность Beacon Messenger

---

## О мессенджере

Beacon — частный мессенджер с открытым кодом, работающий на собственном сервере.
Разработан с приоритетом на конфиденциальность и защиту данных.

---

## 1. Шифрование сообщений (Signal Protocol)

**Алгоритм:** X3DH + Double Ratchet + AES-256-GCM — тот же протокол, что использует Signal

Beacon реализует полный **Signal Protocol** — промышленный стандарт криптографии E2E:

### 1.1 Установка сессии — X3DH (Extended Triple Diffie-Hellman)

Когда Алиса пишет Бобу впервые, они договариваются об общем ключе **не передавая его по сети**:

```
Алиса знает только публичные ключи Боба (с сервера).
Боб хранит свои приватные ключи только на устройстве.

Алиса вычисляет:  ECDH(ik_A, spk_B) + ECDH(ek_A, ik_B) + ECDH(ek_A, spk_B) + ECDH(ek_A, opk_B)
Боб вычисляет:    то же самое зеркально — получают одинаковый SessionKey

Сервер видит только публичные ключи — вычислить SessionKey математически невозможно.
```

- `ik` — долгосрочный Identity Key (генерируется при регистрации)
- `spk` — Signed PreKey (ротируется еженедельно, подписан Identity Key)
- `opk` — One-Time PreKey (используется один раз, затем уничтожается)
- `ek` — Ephemeral Key (одноразовый ключ Алисы, создаётся на каждую сессию)

### 1.2 Шифрование сообщений — Double Ratchet

После установки сессии каждое сообщение шифруется **уникальным ключом**, который никогда не повторяется:

**Симметричный рэтчет** (цепочка ключей):
```
ChainKey₀  →  MessageKey₁ (сообщение 1)
ChainKey₁  →  MessageKey₂ (сообщение 2)
ChainKey₂  →  MessageKey₃ (сообщение 3)  ...и так далее
```
Каждый MessageKey используется один раз и удаляется. Цепочка необратима — из MessageKey₃ не получить MessageKey₁.

**DH-рэтчет** (смена корневого ключа):

Каждое сообщение несёт новый публичный EC-ключ отправителя. При его получении оба устройства выполняют новый ECDH и полностью обновляют все цепочки ключей. Это обеспечивает **"break-in recovery"** — восстановление после компрометации.

### 1.3 Что это даёт на практике

| Свойство | Значение |
|----------|----------|
| **Прямая секретность** (Forward Secrecy) | Взлом текущего ключа не раскрывает прошлые сообщения — они зашифрованы другими ключами, уже удалёнными |
| **Восстановление после взлома** (Break-in Recovery) | Даже если злоумышленник получил текущий ключ, следующий DH-шаг сгенерирует полностью новые ключи — и он снова «выпадет» из переписки |
| **Уникальный ключ на сообщение** | Дешифровка одного сообщения не помогает дешифровать остальные |
| **Одноразовые PreKeys** | Один OPK = одна сессия. После использования уничтожается, повторить невозможно |

> Даже если сервер будет взломан или приватный ключ устройства скомпрометирован — злоумышленник получит доступ максимум к нескольким сообщениям текущей DH-эпохи, но не ко всей переписке.

### 1.4 Технические параметры

- Эллиптическая кривая: **secp256r1 (P-256)**
- Шифрование: **AES-256-GCM** (аутентифицированное шифрование с тегом целостности)
- KDF: **HKDF-SHA256** (RFC 5869)
- Симметричный рэтчет: **HMAC-SHA256**
- Приватный Identity Key защищён **Android KeyStore** (аппаратный чип)

## 2. Шифрование групповых чатов

**Алгоритм:** AES-256-GCM (отдельный ключ на каждую группу)

- Групповой ключ шифруется индивидуально для каждого участника через ECDSA
- При удалении участника — **автоматическая ротация ключа**: старые сообщения недоступны новым участникам, удалённый — не читает новые
- Ключ распределяется зашифрованным, сервер его не знает

---

## 3. Шифрование звонков (E2E)

**Алгоритм:** DTLS-SRTP (стандарт WebRTC)

- Голос и видео шифруются end-to-end между устройствами
- Ключи генерируются на устройствах, сервер их не видит
- Даже TURN-сервер (ретранслятор при NAT) получает только зашифрованный трафик
- При использовании TURN-ретрансляции содержание звонка **недоступно** владельцу сервера

---

## 4. Защита локального хранилища

**Алгоритм:** AES-256-GCM (EncryptedSharedPreferences)

- Все данные на устройстве (сообщения, ключи, контакты) хранятся в зашифрованном виде
- Ключ шифрования защищён Android KeyStore — аппаратным хранилищем ключей
- На устройствах с Secure Enclave / StrongBox — ключ хранится в изолированном чипе

---

## 5. Резервное копирование

**Алгоритм:** AES-256-GCM + PBKDF2 (100 000 итераций)

- Бэкап зашифрован паролем пользователя
- 100 000 итераций PBKDF2 делают брутфорс пароля крайне медленным
- Без пароля файл бэкапа бесполезен

---

## 6. Аутентификация пользователя на сервере

**Алгоритм:** Challenge-Response ECDSA

- При подключении сервер отправляет случайный «вызов» (challenge)
- Клиент подписывает его своим приватным ключом
- Сервер проверяет подпись через публичный ключ
- Пароль по сети **никогда не передаётся**

---

## 7. Инвайт-коды

**Алгоритм:** ECDSA-подпись + TTL 7 дней

- Каждый инвайт-код подписан приватным ключом создателя
- Подпись проверяется при добавлении контакта — подделать код нельзя
- Код действует 7 дней, после истечения — отклоняется

---

## 8. Одна сессия на устройство

- Каждое устройство имеет уникальный `device_id`
- При входе с нового устройства — старая сессия автоматически завершается
- Пользователь получает уведомление о новом подключении

> Нельзя незаметно подключиться к аккаунту с другого устройства.

---

## 9. Транспортная безопасность

- Соединение с сервером: **WSS (WebSocket Secure)** — TLS 1.3
- nginx принимает соединения с Let's Encrypt сертификатом
- Поддержка **certificate pinning** (привязка к конкретному сертификату сервера)

---

## 10. Сопротивление цензуре и DPI

- Сервер маскируется под HTTPS трафик Microsoft на порту 443
- Встроенная поддержка **Tor (Orbot)** для полной анонимизации трафика
- При использовании Tor — сервер не знает реальный IP-адрес клиента

---

## 11. Экстренная защита

**Паник-пароль (Panic Password)**

- Специальный пароль, при вводе которого все данные мгновенно уничтожаются
- Удаляются: сообщения, ключи, контакты, настройки
- Защита при физическом принуждении

## Чего сервер НЕ знает

| Данные | Доступность серверу |
|--------|---------------------|
| Текст сообщений | ❌ Зашифровано E2E |
| Медиафайлы | ❌ Зашифровано E2E |
| Содержание звонков | ❌ Зашифровано DTLS-SRTP |
| Приватные ключи | ❌ Только на устройстве |
| Пароль пользователя | ❌ Только хэш на устройстве |

**Сервер видит:** кто с кем общается (метаданные), время и объём сообщений.
При использовании Tor — метаданные тоже скрыты.
